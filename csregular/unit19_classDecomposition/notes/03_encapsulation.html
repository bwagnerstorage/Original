Encapsulation

In the last lesson we learned that encapsulation is the process of combining variables and methods in a single package called a class. Another aspect of encapsulation is the principle of information hiding. With information hiding objects hide their data (instance variables) from other classes by allowing access to the data only through the methods defined within the class. Encapsulation therefore guarantees the integrity of the data contained in the object by controlling how the data is accessed and modified.

Television Example

To help illustrate this point let's look at a real world example - a television. How do you interact with a TV? You either use a remote or the buttons provided on the front of the console. The manufacturer has provided you with an interface(remote and buttons) to make using the TV easier. Any interaction with the TV occurs through this interface.

Do you need to know how a TV works inorder to use it? No, all you need to know is how to turn it on, change the channel, or increase or decrease the volume. In fact, the manufacturer has hidden the details of how the TV works from you by enclosing it in a case. It would not be advisable for you to remove the case and mess with the TV's internal circuitry.

Just like a TV manufacturer, a programmer, when designing a class, provides an interface. The interface consists of the public methods defined within the class. Interaction with the class can only take place through these provided methods. The instance variables are the internal circuitry of the class. Just as it is not wise to mess with a TV's internal circuitry, it is not advisable for programmers to have direct access to the data within a class. This packaging or encapsulating the data and methods of an object together provides protection from external tampering by users. Many possible errors can be avoided by adhering to this programming principle.

Benefits of Encapsulation

  • It simplifies the interaction between objects.
  • Allows a class to hide the internal details of how an object does something.
  • A class can change its internal implementation without hurting the overall functionality of the system.
  • Prevents code (data) from accidental corruption due to programming errors.

Visibility Modifiers

In Java, we create object encapsulation using visibility modifiers. They control whether other classes can "see" what's "inside" an object. The reserved words public and private are visibility modifiers that can be applied to the variables and methods of a class. If a member of a class has public visibility, it can be "seen" (read and perhaps changed) from outside the object. If a member of a class has private visibility, it can be used anywhere inside but not outside the class definition. Below I have highlighted the visibility modifiers used in the Die class.

public class Die
{
    // variable declarations (instance variables)
    
private
 int faceValue;
    
    // method declarations
    
public
 void roll()
    {
        faceValue = (int)(Math.random() * 6) + 1;
    }
    
    
public
 int getFaceValue()
    {
        return faceValue;
    }
}

Public variables let code outside the class reach in and read or change the value of the data. This violates the encapsulation principle. Therefore instance data should be defined with private visibility.

private
 int faceValue;

Data that is declared as private can be read or changed only by the methods of the class, which makes the objects created from that class self-governing. Whether a method has public or private visibility depends on the purpose of that method. Methods that provide services to other classes must have public visibility.

public
 void roll()
{
   faceValue = (int)(Math.random() * 6) + 1;
}
    

public
 int getFaceValue()
{
   return faceValue;
}

These methods are sometimes called service methods. A private method cannot be invoked from outside the class. The only purpose of a private method is to help the other methods of the class do their job. Therefore they are sometimes called support methods.

Constants
